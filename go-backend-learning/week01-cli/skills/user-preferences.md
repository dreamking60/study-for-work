# Skills — 用户偏好与协作流程

## 沟通与语气
- **语言**：优先使用中文回复，只有在用户明确要求其他语言时再切换。
- **确认方式**：用户常用“好，然后呢”“可以...吗”之类的问法，希望先快速确认，再进入执行步骤。
- **信息密度**：答案需要开门见山，先说完成了什么，再补细节与下一步建议。

## 输出格式偏好
- **路径要求**：涉及文档/产物时，要标明绝对路径或至少到 `study-for-work/...` 的完整层级，避免只写相对路径。
- **文件引用**：在说明修改内容时，使用 ``路径:行号`` 的形式，方便快速定位。
- **结构**：倾向使用简短小标题 + 列表。对于简单确认可以直接一句话。

## 常用提问/需求类型
- “请确认项目能否运行” → 期待执行 `go test ./...` 或 `go run` 并反馈结论。
- “把 week/day 的计划细化” → 要求按天提供目标、产出、实施步骤、调试命令和 hints。
- “感觉某内容还不够” → 需要在原有基础上增量补充（例如路径调整、Go 速查扩写）。

## 工作流程（Go 项目）
1. **先验证环境**：运行 `go test ./...` 或相关命令，确认新环境可执行。
2. **分析现有结构**：使用 `rg --files`、`sed -n` 等快速了解项目内容。
3. **制定或调整计划**：必要时使用 plan 工具，列出步骤并在执行后更新状态。
4. **产出文档**：对周计划/日计划使用 `study-for-work/learning-plan/...` 目录，并写明命令、测试与提示。
5. **附加参考**：为离线场景准备 Go 标准库示例或脚本，集中在 `go-recipes.md` 等文件。

## 验收与交付
- 输出结尾附带“后续建议/下一步”提示，帮助用户接着做事。
- 若某命令无法执行（如权限、无网络），要说明原因并提出替代方案。
- 遇到歧义或目录要求（如路径指定）优先满足用户显式偏好。
